You are a senior full-stack engineer building "AutoViva AI" — a production-grade SaaS platform 
that acts as an AI-powered project evaluation agent for college students and hackathon judges.

The platform automates browser-based evaluation of student localhost web projects, executes 
predefined test cases, captures evidence (screenshots, logs, DOM snapshots), and generates a 
professional evaluation report — all within 10 minutes.

---

### CORE CONCEPT

A student runs their project locally (e.g., http://localhost:3000). They open AutoViva AI, 
enter the project URL + credentials + test cases in JSON format. The system:
1. Launches a Chrome browser via PinchTab API
2. Logs into the student's app
3. Executes each test case step-by-step using an AI agent
4. Captures screenshots and evidence after every step
5. Verifies actual result vs expected result
6. Generates a professional evaluation report with a score out of 100

### AGENT PIPELINE (AgentScope Multi-Agent)

Build 7 agents that run in sequence per evaluation:

**Agent 1: ProjectUnderstandingAgent**
Input: project metadata (name, URL, description, tech stack)
Output: { main_modules, risk_areas, suggested_test_coverage }
LLM: Groq LLaMA 3.3 70B

**Agent 2: TestPlannerAgent**
Input: raw test cases (JSON) + project understanding
Output: browser_plan[] for each test case
Each step has: { action, target, value, wait_for, expected_change }
LLM: Groq LLaMA 3.3 70B

**Agent 3: BrowserExecutorAgent**
Uses: PinchTab HTTP API
Actions: navigate, click, fill, wait, screenshot, extract_text, dom_snapshot
After each step: capture screenshot, extract page text, record URL
Does NOT use LLM — pure deterministic browser control

**Agent 4: AssertionAgent**
Input: expected_result + actual page state (text, URL, DOM)
Output: { passed: bool, evidence: {...}, confidence: float }
Rules: NEVER mark passed without evidence. Evidence = screenshot + text match or URL match.
LLM: Groq LLaMA 3.3 70B (for semantic matching only)

**Agent 5: RecoveryAgent**
Triggered when: step fails due to changed UI (button text, route, field label)
Strategy: try up to 2 alternative selectors, then mark as failed with details
LLM: Groq LLaMA 3.3 70B

**Agent 6: ReportAgent**
Input: all test results + evidence
Output: { summary, score, grade, suggestions, ai_notes }
Scoring: use the scoring rubric defined in PRD Section 11
LLM: Groq LLaMA 3.3 70B

**Agent 7: VivaAgent**
Input: full report + Q&A question from examiner
Output: spoken/written answer backed by evidence
Supports voice via Groq Whisper (STT) + Groq TTS
LLM: Groq LLaMA 3.3 70B

### SECURITY RULES (NON-NEGOTIABLE)

1. PinchTab server MUST bind to 127.0.0.1 only
2. All credentials stored AES-256 encrypted in DB
3. Passwords NEVER appear in reports or logs — always masked as ****
4. Browser sessions isolated per project run
5. URL allowlist: localhost, 127.0.0.1, *.local, college internal IPs only
6. JWT tokens expire in 24 hours
7. Rate limit: max 5 concurrent evaluation runs per user
8. Screenshots stored with hashed filenames
9. Browser profile deleted after run completes
10. API endpoints protected with auth middleware

## 1. Product Overview

**Product Name:** AutoViva AI  
**Tagline:** "10-minute project viva? Let the AI examiner test the full project."  
**Type:** SaaS Web Platform (locally installed browser agent)  
**Target Users:** Engineering students, external examiners, hackathon judges, faculty  
**Core Value:** Automated, evidence-based, AI-powered evaluation of student software projects running on localhost  

---

## 2. Problem Statement

In college external project evaluations, students get 5–10 minutes to demonstrate full software systems. External examiners check 1–2 features due to time constraints. This causes three critical problems:

- **Problem 1:** Good projects are under-evaluated. Complex features never get shown.
- **Problem 2:** Fake or incomplete projects can pass if the login screen looks good.
- **Problem 3:** Students cannot prove full functionality without live demos that crash under pressure.

**AutoViva AI solves this by acting as an AI examiner that opens the project, tests every feature, captures proof, and delivers a scored report — all automatically.**

---

## 3. User Personas

### Persona 1: The Student (Primary)
- BE/BTech final year student
- Has a web project running on localhost
- Wants to demonstrate all features fairly
- Needs a polished evaluation report as evidence

### Persona 2: The External Examiner (Secondary)
- Faculty from another college
- Has 5–10 minutes per student
- Cannot deeply verify every feature manually
- Trusts an automated, evidence-backed report

### Persona 3: The Hackathon Judge (Future)
- Evaluates 20–50 projects in one day
- Needs fast, consistent, objective evaluation
- Wants feature proof without sitting through each demo

### Persona 4: The Faculty/HOD (Future)
- Manages all student project evaluations
- Wants ranked output with scores and evidence
- Needs to export and submit reports to university

---

## 4. User Flows

### Student Flow (Primary)
1. Sign up / Log in to AutoViva AI
2. Create a new Project (enter name, URL, description, tech stack)
3. Add Credentials (email, password, role)
4. Open Test Suite Builder — import JSON or build manually
5. Choose Evaluation Mode (Pre-check / Live Demo / Strict)
6. Click "Run Evaluation"
7. Watch live browser execution on screen
8. Review the final report
9. Download PDF report
10. Show report to examiner during viva

### Examiner Flow
1. Student opens AutoViva AI during viva
2. Examiner asks student to run full evaluation
3. Examiner watches browser agent execute tests in real time
4. Examiner reads final report + score
5. Examiner asks voice/text questions to Viva Agent
6. Examiner accepts the evaluation report as proof

---

## 5. Evaluation Modes

| Mode | Who Uses It | Browser | Voice | Intervention |
|------|-------------|---------|-------|--------------|
| Pre-check | Student before viva | Headless | No | Recovery allowed |
| Live Demo | Viva with examiner | Headed (visible) | Yes | None |
| Strict | Faculty/Hackathon | Headed | No | None |
| Quick Smoke | Fast sanity check | Headless | No | Recovery allowed |

## 10. PinchTab Integration Reference

PinchTab exposes a local HTTP API. Use these endpoints in BrowserExecutorAgent:

```python
# Start a browser session
POST http://127.0.0.1:9000/api/sessions
{ "profile": "autoviva-run-{run_id}", "headless": false }
Response: { "session_id": "abc123" }

# Navigate
POST /api/sessions/{session_id}/navigate
{ "url": "http://localhost:3000/login" }

# Take snapshot (structured content)
GET /api/sessions/{session_id}/snapshot
Response: { "text": "...", "interactive_elements": [...] }

# Click element
POST /api/sessions/{session_id}/click
{ "ref": "element_ref_from_snapshot" }

# Fill input
POST /api/sessions/{session_id}/fill
{ "ref": "element_ref", "value": "admin@test.com" }

# Screenshot
GET /api/sessions/{session_id}/screenshot
Response: { "image_base64": "..." }

# Close session
DELETE /api/sessions/{session_id}
```

## 11. Anti-Hallucination Rules (Agent Constraint Layer)

A test case is ONLY marked as PASSED if at least ONE of the following is true:

1. `expected.url_contains` matches current URL
2. `expected.page_contains` text found in extracted page text
3. `expected.element_visible` reference found in DOM snapshot
4. Success message matching expected string is visible

A test case is marked FAILED if:
- Browser throws uncaught error
- Expected condition not met within timeout
- Recovery agent exhausts retry limit
- Network request returns 4xx/5xx status

Every PASSED result in the report must link to:
- At least one screenshot
- The page text excerpt showing expected content
- The URL at time of assertion

## 14. Deployment Architecture (Production)

```
Frontend:  Vercel (Next.js)
Backend:   Railway or Render (FastAPI + Celery)
Database:  Supabase (PostgreSQL)
Redis:     Railway Redis or Upstash
Storage:   Cloudflare R2 (screenshots/reports)
Auth:      Supabase Auth or custom JWT
Domain:    autoviva.ai (suggested)
```

**Note:** PinchTab browser agent runs LOCALLY on the student's machine. The frontend/backend can be hosted in cloud, but the browser controller must run locally where the project runs on localhost.

**Local Agent Setup for Students:**
```bash
# Student installs AutoViva Agent CLI
npm install -g autoviva-agent

# Start local agent (connects to cloud backend)
autoviva-agent start --token YOUR_USER_TOKEN

# Agent registers itself → backend knows browser is ready → runs start
```
